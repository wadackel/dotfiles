# Scenario Design and Subagent Launch Contract

Single source of truth for designing evaluation scenarios, building the subagent invocation prompt, the dry-run contract, and the hold-out scenario generation rule. Referenced from SKILL.md Step 3 (Scenario Design), Step 4 (Dispatch), and Step 8 (Convergence + Hold-out).

---

## Scenario Design

### How many scenarios

- **Minimum 2, recommended 3**: 1 median (typical use) + 1-2 edges (boundary or unusual conditions).
- 1 scenario overfits — passing 1 scenario tells you nothing about generalization.
- More than 3 inflates per-iteration cost without much information gain. Add a 4th only if the skill has truly distinct workflow modes.

### What makes a good scenario

- **Reflects real use**: pulled from situations the target skill will actually encounter, not synthetic puzzles.
- **Distinct from siblings**: each scenario should exercise different code paths / decision branches in the skill. If two scenarios produce the same execution trace, drop one.
- **Self-contained**: the subagent must be able to act on the scenario from a one-paragraph setup. No "ask the user" loops.

### Requirements checklist (per scenario)

For each scenario, list **3-7 requirements** the deliverable must satisfy. Format:

```
1. [critical] <minimum-bar item>
2. <regular item>
3. <regular item>
...
```

Rules:
- **At least 1 `[critical]` tag per scenario, mandatory**. Without `[critical]` the success/fail axis becomes vacuous (everything is "partially passing").
- Keep requirements observable: "the output contains a section titled 'Risks'" beats "the output is well-organized".
- Score: success = ALL `[critical]` items are ○. Even one × on a `[critical]` item = failure.

### Frozen after agreement

- Once the user signs off on scenarios + checklists, **freeze them**. Do not edit mid-run.
- Editing scenarios mid-iteration invalidates cross-iteration comparison (you no longer know whether the metric change came from the patch or the moved goalpost).
- If a scenario turns out to be wrong, **stop the run, redesign, restart from iteration 1**.

### How Claude proposes scenarios (default UX)

1. Read the target skill's `description` and body to understand its scope.
2. Propose 2-3 scenarios + per-scenario requirements checklist as a single message.
3. Wait for user feedback. User can edit, add, drop, or fully rewrite.
4. On user confirmation: freeze. Do not propose alternatives later.

## Subagent Launch Contract

The prompt template that Claude sends to each freshly-dispatched subagent via the **Agent tool** (`subagent_type: "general-purpose"` by default). Mirrors the empirical-prompt-tuning launch contract.

### Template

```
あなたは <target-skill-name> を白紙で読む実行者です。

## 対象スキル
<full body of the target SKILL.md, OR an absolute path the subagent should Read>

## シナリオ
<one-paragraph scenario setup from Scenario Design>

## 要件チェックリスト（成果物が満たすべき項目）
1. [critical] <minimum-bar item>
2. <regular item>
3. <regular item>
...

判定規則: [critical] 項目が **全て ○** のときのみ成功（○）。1 つでも × または 部分的なら失敗（×）。

## タスク
1. 対象スキルの指示に従ってシナリオを実行し、成果物を生成する。
2. 終了時に下記レポート構造で返答する。

## レポート構造
- **成果物**: <生成物 / 実行結果サマリ>
- **要件達成**: 各項目について ○ / × / 部分的（理由付き）
- **不明瞭点**: 対象スキルで詰まった箇所、解釈に迷った文言（箇条書き）
- **裁量補完**: 指示で決まっておらず自分の判断で埋めた箇所（箇条書き）
- **再試行**: 同じ判断をやり直した回数とその理由

## Dry-run convention
成果物 (実装案 / 実行する command 列 / 出力サンプル) のみを返してください。
- 実 file 編集を **行わない**
- 外部 API 呼び出し / git commit / 環境への副作用を **行わない**
- 「実行するならこの command」を明記し、実行はしない
副作用を完全に避けられない skill (e.g. 完全に dry-run できない destructive な性質) は本評価の対象外です。
```

### Parallel scenarios

Multi-scenario evaluation: dispatch all scenarios in **a single message containing multiple Agent tool calls**. The Agent tool runs them in parallel and returns each subagent's report along with its `tool_uses` and `duration_ms` usage metadata.

### Never reuse a subagent

Always dispatch fresh. A reused subagent has implicit memory of prior iterations' patches and will under-report unclear points it has already worked through. Bias is invisible but real.

## Hold-out Scenario Generation

Generated by Claude **immediately before** the convergence verdict (after the second consecutive clean iteration, before declaring CONVERGED).

### Why generate, not pre-allocate

A truly held-out scenario from Step 3 would sit unused for several iterations and might no longer reflect the skill's current shape. Generating fresh from the frozen scenarios at convergence time keeps the test relevant.

### Generation rule (avoids fit-to-pass bias)

For each frozen scenario, identify its **dimensions**:
- **Input size**: number of items, length of text, file count
- **Domain**: subject area, file type, language
- **Format**: structured / unstructured, single / batched, sync / async

Pick **one** frozen scenario at random. Shift **one of its dimensions by 1 degree** (e.g., size 5 items → 25 items; domain "Python codebase" → "TypeScript codebase"; format "single PR" → "stacked PRs"). Hold the other dimensions constant.

This produces a scenario that:
- Exercises the same skill capabilities the iterations tuned for
- Is novel enough to detect overfitting to the specific frozen examples

### Hold-out judgement

Run hold-out using the same launch contract. Compare its accuracy to the average of the last 2 frozen-scenario iterations:
- **Drop ≤ 15 percentage points** → generalization holds, declare CONVERGED
- **Drop > 15 percentage points** → overfit detected. Allow **1 additional iteration** to address the hold-out's unclear points, then re-test ONCE. If still > 15 pt drop, declare DIVERGED and recommend redesigning scenarios (the frozen scenarios were too narrow).

Do not run hold-out more than twice. Beyond that, you are tuning to the hold-out, defeating the point.
